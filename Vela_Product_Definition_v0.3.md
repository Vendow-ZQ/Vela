> 版本：V0.3  
> 日期：2026-05-20

## 0. 当前阶段判断

Vela 已经从“AI 健身硬件 / 健康穿戴”的模糊概念，收敛为：

> **一个以 App MVP 先行的 AI 健身 Agent，通过用户结构化反馈、训练模板、规则系统和 LLM 交互层，先跑通训练日闭环和组间动态调参；未来再接入第三方健康数据、自有腕带、Wrap / Pro Sensor 和视觉系统增强判断精度。**

当前阶段的核心决策已经完成：

- 产品哲学：主动训练决策系统
- MVP 形态：Web App 先行
- 主战场：健身房力量训练
- 核心体验：组间动态调参
- 用户权力：用户拥有训练主导权，Vela 拥有专业建议权
- Agent 原则：正常时隐身，异常时介入
- 训练流：完整训练日闭环，重点展示卧推 / Push Day 的组间调参
- 数据原则：记录计划、实际、体感、调整、用户主动修改、Agent 自动调整与 Memory

下一阶段应进入：

1. Web App PRD 正式整理
2. 页面低保真 / 中保真原型
3. Demo 剧本与 Mock 数据
4. Agent 规则表
5. 前端开发任务拆解
6. Wrap / Sensor 概念线与 App 数据接口对齐

## 1. 一句话定义

**Vela 是一个 AI 健身 Agent：基于用户的生理数据、训练数据和实时训练质量，主动判断今天怎么练、这一组练得怎么样、下一组怎么调。**

更具产品表达的一句话：

**Vela = 你的健身房私教，也是你的日常健康托底。**

更底层的一句话：

**Vela 是一个主动训练决策系统。**

---

## 2. 第一性产品本体

Vela 的第一性产品本体不是：

- 腕带
- 手表
- 手套
- 摄像头
- 体脂秤
- 健康数据 App
- 训练记录 App
- 动作百科 App

Vela 的第一性产品本体是：

> **主动训练决策系统。**

它要持续回答三个问题：

1. **今天该怎么练？**
2. **这一组练得怎么样？**
3. **下一组该怎么调？**

硬件、传感器、第三方健康数据、体测报告、摄像头、腕带、Wrap / Sensor，都是这个训练决策系统的数据入口或体验增强器。

---

## 3. 产品哲学

### 3.1 主动指导是核心，被动监测是支撑

Vela 不以“健康数据展示”为中心，而以“训练决策”为中心。

睡眠、HRV、心率、体脂、体重、恢复状态、久坐数据等，都不是为了形成健康大屏，而是为了回答：

- 今天适不适合冲重量？
- 热身要不要更保守？
- 下一组要不要加重？
- 这节课是否应该降容量？
- 本周训练模板是否需要调整？
- 用户是否正在进入高风险训练状态？

### 3.2 稳定进步 + 少受伤

Vela 的长期价值不是让用户每次训练都爽，而是帮助用户：

- 更稳定地进步
- 更少瞎练
- 更少伤病
- 更理解自己的训练状态
- 更长期地保持训练能力

表层获客诉求可以是增肌、塑形、减脂、增力、力形兼备；长期留存价值是更科学训练、更少受伤、更少走弯路。

### 3.3 正常时隐身，异常时介入

Vela 不应该为了显得智能而频繁说话。

正常训练时：

- 不打扰
- 不解释
- 不弹建议卡
- 按计划推进

出现偏差或风险时：

- 介入
- 调整
- 兜底
- 必要时明确反对

Vela 的智能不体现在每组都说话，而体现在 **该不说话时不说，该介入时介入**。

### 3.4 用户拥有训练主导权，Vela 拥有专业建议权

Vela 是辅助型智能体，不是训练控制器。

用户决定：

- 是否开始训练
- 是否执行计划
- 是否修改动作
- 是否接受建议
- 是否结束动作
- 是否结束训练
- 是否调整今日计划

Vela 负责：

- 生成默认计划
- 给出组间调参建议
- 识别异常和风险
- 提供替代方案
- 记录训练变化
- 在必要时明确反对高风险行为

一句话：

> **用户是训练主导者，Vela 是智能副驾。**

---

## 4. 目标用户

MVP 首发用户是：

> **中阶健身房用户。**

典型画像：

- 训练 6-24 个月，或 6 个月-3 年
- 会基本动作
- 知道卧推、深蹲、硬拉、PPL、增肌、减脂、RPE 等概念
- 不想长期买私教
- 也不想靠网上视频和碎片化攻略瞎练
- 目标覆盖增肌、塑形、减脂、增力、力形兼备
- 真正缺的是：有人告诉他“下一组怎么练”

MVP 不优先服务：

- 纯户外跑步用户
- 纯瑜伽 / 普拉提用户
- 专业运动员
- 45+ 慢病管理用户
- 只想看健康数据、不做力量训练的人
- 极高阶力量举 / 健美用户

## 5. 产品路线

### 5.1 MVP 0：Web App / App Agent

第一阶段先做 Web App，而不是先做硬件。

Web App MVP 验证的是：

- 用户是否理解 Vela 是 AI 健身 Agent
- 用户是否愿意按 Vela 的计划进入训练
- 用户是否愿意每组后做极轻量反馈
- Vela 是否能根据反馈给出可信调参
- 用户是否觉得它比 Fitbod / Keep / 自己瞎练更像私教
- 训练后复盘是否让用户觉得“它懂我刚刚练得怎么样”

### 5.2 MVP 1：App + 第三方健康数据

第三方健康数据是增强项，不是主链路依赖。

可接入：

- Apple Health
- Google Fit
- 华为健康
- 小米健康
- 手环 / 手表历史数据
- 体测报告
- 体脂秤数据

没有第三方数据，Vela 仍然可以运行；有数据，恢复判断和长期趋势更稳。

### 5.3 V1：Vela Band

腕带不是本体，而是 Vela Agent 的身体状态入口。

它增强：

- 心率
- HRV
- 睡眠
- 训练提醒
- 组间休息
- 震动反馈
- 小屏确认
- 基础 IMU
- 日常恢复状态托底

### 5.4 V1 Pro / V2：Vela Wrap / Sensor

未来可探索：

- 可卷绕训练器械传感带
- 多点压力传感阵列
- IMU
- 握持压力变化
- 左右受力趋势
- 推 / 拉节奏
- 深蹲杠位偏移
- 高位下拉 / 划船两侧发力同步性

暂定概念：

> **Vela Sensor Wrap：可卷绕训练器械传感带。它把普通杠铃、哑铃和器械把手临时变成可感知的训练接触界面，为 Vela Agent 提供更高精度的受力、稳定性和节奏数据。**

但 Wrap 不作为 App MVP 默认入口。

它不应在第一版承诺精准测真实重量、精准做功功率或完整动作纠错。更稳的定位是：

> **训练接触质量传感器。**

## 6. Web App MVP 演示目标

Web App MVP 的演示目标是：

> **跑通完整训练闭环，但重点展示 Agent 组间调参能力。**

主流程完整：

首页 → 开始训练 → 训练前协商 → 今日课表 → 动作训练 → 组后记录 → 组间调参 → 主动结束训练 → 训练复盘

训练内容聚焦：

- Push Day
- 主项：卧推
- 辅项可展示，但不必全量交互完成

核心能力展示：

- 正常完成
- 过轻 → 立即提高训练刺激
- 轻松 → 先观察，连续两组再调整
- 粘滞 → 先观察，连续两组再保守调整
- 失败 → 失败类型确认
- 被压 / 疼痛 / 动作变形 → 高风险兜底
- 替换动作 / 器械被占 → 今日计划动态调整
- 训练复盘 → 自动更新下一次训练计划草稿

## 7. Web App 信息架构

Vela Web App MVP 采用“训练前—训练中—训练后”的内部结构，但前台不直接使用这个命名。

### 7.1 训练前

#### 首页

首页是个人训练资产简化页 + 今日训练入口。

展示：

- 今日 Agent 建议
- 今日计划卡
- 核心身体数据
- 三大项预测 1RM / e1RM
- 开始训练按钮
- 个人数据页入口
- 训练记录 / 日历页入口

#### 个人数据页

个人数据页是详细训练资产页。

展示：

- 身体数据趋势
- 力量画像
- 训练质量 Agent 总结
- 未来趋势 / 下一阶段建议

### 7.2 训练中

#### 协商页

点击开始训练后进入。

采用分步问答：

1. 今日状态
2. 可训练时间
3. 不适 / 限制
4. Vela 预判今日策略，用户确认或改写

#### 今日计划页

展示今日动态训练计划。

包括：

- 训练主题
- 今日策略：稳 / 冲 / 保
- 动作列表
- 动作角色
- 组数 / 次数区间
- 今日重点
- 替换 / 器械占用入口
- 增加 / 删除 / 替换 / 调整顺序

#### 动作组数重量页

核心训练执行页。

默认只展示当前组信息，不展示复杂功能面板。

包括：

- 当前动作
- 当前组数
- 本组计划重量 × 次数
- 体感滑动控件
- 完成按钮
- 休息倒计时
- 查看计划
- 替换动作

### 7.3 训练后

#### 日历记录页

采用月历视图。

训练日用圆圈标记，圆圈内显示当天训练容量，颜色越深表示容量越高。

点击日期后展示简短训练摘要，并可进入详细训练复盘页。

#### 训练复盘页

训练复盘页既可以从训练结束后进入，也可以从日历记录页独立进入。

采用三层信息结构：

1. 当天最简训练总结
2. 每个动作的汇总卡
3. 展开某个动作后查看每组记录

## 8. 首页定义

首页是 **个人训练资产简化页 + 今日训练入口**。

首页展示：

### 8.1 今日 Agent 建议

示例：

> 今天计划：Push Day。建议先完成卧推主项，点击开始训练后我会根据你的状态、时间和不适重新调整。

注意：首页不直接询问稳 / 冲 / 保。稳 / 冲 / 保发生在点击开始训练后的训练前协商中。

### 8.2 今日计划卡

示例：

- Push Day
- 主项：卧推
- 预计：45-60 分钟
- 重点：胸部主项 + 肩三头容量

这是预告，不是最终课表。最终课表由训练前协商后生成。

### 8.3 核心身体数据

展示：

- 体重
- 体脂
- 肌肉量

首页只显示简化值。详细趋势进入个人数据页。

### 8.4 三大项预测 1RM / e1RM

展示：

- 卧推 e1RM
- 深蹲 e1RM
- 硬拉 e1RM

e1RM 用于成就感、进步感和训练资产展示。后台调参仍基于稳定工作组、RPE/RIR、历史完成情况、当天状态和组间反馈。

MVP 不做预测 1RM 分享卡片。用户想炫耀，自己截图即可。

### 8.5 主 CTA

按钮：

> 开始训练

## 9. 个人数据页定义

个人数据页不是健康报告，而是 **训练资产与身体变化的详细驾驶舱**。

它不优先展示风险和状态，不做焦虑化提醒。风险信息如果出现，也包装为训练质量或调整建议。

### 9.1 身体数据趋势

MVP 展示：

- 体重
- 体脂
- 肌肉量

展示近 7 天变化。

示例：

> 体重小幅上升，同时肌肉量和训练表现都在提升，当前更接近稳定增肌趋势。

不表达：

> 你胖了 / 体脂偏高。

### 9.2 力量画像

力量画像采用 **Agent 结论优先，具体动作数据下钻**。

MVP 先按五个部位组织：

- 胸部力量
- 背部力量
- 肩部力量
- 手臂力量
- 腿部力量

第一层由 Vela 给自然语言判断：

- 哪些是优势项
- 哪些稳定
- 哪些需要增强
- 哪些短板需要补齐

示例：

> 你的胸部力量是当前最稳定的优势项，卧推完成度不错；腿部力量基础较好，但最近缺少更新；背部力量是当前短板，本周 Pull Day 缺失，建议下次优先补一次拉类训练。

展开后展示具体数据：

- e1RM
- 稳定工作组
- 最近一次训练数据
- 体感分布
- Vela 判断

### 9.3 训练质量

训练质量模块不是指标面板，而是 Vela 对一段时间训练表现的自然语言评估。

MVP 默认总结近 7 天。

示例：

> 这周你完成了 2 次训练，Push Day 表现比较稳定，卧推主项完成度不错，但后两组出现粘滞。Pull Day 暂时缺失，背部训练容量偏少。

展开后可查看依据：

- Push / Pull / Legs 完成情况
- 训练容量
- 体感分布
- 粘滞 / 失败记录
- 动作完成度

长期方向支持 7 天 / 30 天 / 90 天切换；MVP 先做 7 天。

### 9.4 未来趋势 / 下一阶段建议

MVP 做自然语言趋势建议，不做复杂预测图。

示例：

> 如果保持当前每周 2-3 次训练频率，卧推可以先把 60kg × 8 的后两组从“粘滞”变成“正常”。在这个基础上，再考虑进入 62.5kg 的小幅加重。

不做精密预测：

> 你 6 周后一定卧推 100kg。

## 10. 训练前协商

点击“开始训练”后，进入训练前协商。

训练前协商采用 **分步问答 + 自然语言 / 卡片 / 自由输入混合**。

### 10.1 顺序

1. 今日状态
2. 可训练时间
3. 不适 / 限制
4. Vela 预判今日策略，用户确认或改写

### 10.2 交互形式

每一步上方由 Vela 用自然语言提出问题，下方提供结构化选项按钮，同时保留自由输入入口。

示例：

Vela：

> 开始前先确认一下，今天身体状态怎么样？

选项：

- 轻松兴奋
- 正常
- 有点困 / 疲劳
- 自己说一句

### 10.3 稳 / 冲 / 保

稳 / 冲 / 保不是首页入口，而是训练前协商后的策略标签。

- **稳**：按计划推进
- **冲**：尝试突破
- **保**：降低风险，保住训练节奏

Vela 先根据计划、历史、状态、时间、不适预选策略，用户可以修改。

### 10.4 用户坚持冲但状态不支持

如果只是状态一般：

- 允许以热身校验作为前置条件

如果存在疼痛、高风险或明显不适合：

- Vela 明确反对冲当前动作
- 给替代突破目标，如冲动作质量、冲稳定次数、冲训练完成度或换低风险动作

## 11. 今日计划页

今日计划页展示完整但简洁的动态训练计划。

每个动作卡片包括：

- 动作名称
- 动作角色：主项 / 主要辅项 / 孤立 / 收尾
- 组数 / 次数区间
- 预计休息
- 今日重点
- 替换入口
- 器械占用入口

示例：

**卧推｜主项**  
4 组 × 5-8 次  
休息 2-3 分钟  
今日重点：稳定完成，不冲极限  
入口：替换｜器械被占

今日计划页支持用户编辑：

- 增加动作
- 删除动作
- 替换动作
- 调整顺序

但编辑能力保持轻量，不做长期模板编辑。

### 11.1 用户主动编辑原则

用户可以改今日计划。

Vela 不阻止编辑，但会在编辑影响训练目标、安全或主项表现时给出轻提醒。

如果用户坚持，Vela 接受，并在训练中按新计划继续辅助。

---

## 12. 动作组数重量页

动作组数重量页是 Web App MVP 的核心页面。

### 12.1 当前组聚焦

页面默认只展示当前组情况。

不展示：

- 大段解释
- 所有功能按钮
- 全部组记录
- 复杂数据面板

默认结构：

```text
卧推｜主项｜第 2 / 4 组

本组计划
60kg × 8

体感
← 过轻  轻松  正常  粘滞  失败 →
              正常

[完成]

休息中
01:25

[查看计划]        [替换动作]
```

### 12.2 不需要点击开始本组

App MVP 不要求用户每组开始前点击“开始本组”。

真实训练里，用户自然记录的时刻是做完一组之后。

流程：

```text
Vela 给出本组计划
  ↓
用户训练
  ↓
组后记录页预填计划
  ↓
用户一键确认，或修改实际完成情况
  ↓
Vela 生成下一组计划或调整
```

### 12.3 默认正常，一键完成

组后记录页预填：

- Vela 建议重量
- Vela 建议次数
- 默认体感：正常

如果用户按计划完成，点击完成即可提交。

系统记录：

- 重量 = Vela 计划重量
- 次数 = Vela 计划次数
- 体感 = 正常
- 异常 = 无

然后直接进入下一组，开始休息计时。

### 12.4 修改重量 / 次数

主页面不额外放“修改”按钮。

本组计划数字本身可点击。用户点击 `60kg × 8` 后，弹出轻量编辑面板：

- 实际重量
- 实际次数

确认后回到当前组主界面。

### 12.5 体感状态

前台体感词固定为：

- 过轻
- 轻松
- 正常
- 粘滞
- 失败

默认停在“正常”。

含义：

- **过轻**：计划明显低估强度，需要立即调整
- **轻松**：有余力，但先观察，连续两组轻松后再调整
- **正常**：计划匹配，不打扰，按计划继续
- **粘滞**：完成了，但最后几次明显卡住；连续两组粘滞后进入保守调整
- **失败**：没有完成或出现安全事件，需要二次确认失败类型

### 12.6 正常完成不打扰

当用户按计划完成本组，且体感为正常时：

- 不弹建议卡
- 不解释
- 不调整重量和次数
- 直接按原计划进入下一组
- 开始休息计时

原则：

> **正常完成 = 按原计划继续，不打扰用户。**

### 12.7 状态调整规则

立即调整类：

- 过轻
- 失败
- 被压
- 疼痛

观察后调整类：

- 轻松
- 粘滞

规则：

- 过轻：立即提高训练刺激
- 轻松：先记录，连续两组轻松后调整
- 正常：继续
- 粘滞：先记录，连续两组粘滞后调整
- 失败：进入失败类型弹窗

## 13. 查看计划入口

动作组数重量页里的“查看计划”入口，打开的是当前训练的今日计划页视图，而不是一个新的计划弹层。

训练中查看计划时：

- 当前休息倒计时继续
- 页面顶部或底部保留小型倒计时悬浮条
- 用户可以随时返回当前组

示例：

```text
卧推 第 2 / 4 组休息中｜剩余 01:12｜返回当前组
```

## 14. 替换动作入口

动作组数重量页底部只保留两个辅助入口：

- 查看计划
- 替换动作

失败、疼痛、不适、被压等组内异常不单独做按钮，而是通过体感状态滑动到“失败”后进入失败类型弹窗。

器械被占、想换动作、动作不舒服、直接结束动作等动作级问题统一从“替换动作”入口进入。

### 14.1 替换动作第一层

文案保持简练：

```text
调整原因

器械被占
做着不舒服
换个类似动作
结束这个动作
其他
```

### 14.2 器械被占

选项：

- 可轮流做
- 等 5-10 分钟
- 要等很久 / 不想等
- 自己说一句

逻辑：

**可轮流做**  
保留当前动作，不替换主项。休息时间跟随实际轮流节奏，必要时自动放宽间歇。

**等 5-10 分钟**  
不直接换动作，建议做低疲劳准备，不提前消耗主项。

**要等很久 / 不想等**  
Vela 提供替代方案或临时改主项，但需要用户确认。

**自己说一句**  
用户自由输入，Vela 再从替代池和当前训练目标里给方案。

### 14.3 换个类似动作

Vela 先推荐一个最佳替代，再提供少量替代选项和自由输入。

统一交互原则：

> **Vela 先给专业推荐，用户再选择、改写或自由输入。**

示例：

```text
建议替换为：哑铃卧推

原因：它和卧推动作模式接近，不需要卧推架，今天仍然可以作为胸部主项。

按这个来
换成器械推胸
换成史密斯卧推
自己说一句
```

### 14.4 结束这个动作

用户点击“结束这个动作”后，系统直接结束当前动作，不强制追问原因。

后台根据完成进度判断：

- 已完成 ≥ 70% 计划组数：直接结束，进入下一动作
- 已完成 30%-70%：直接结束，但记录为提前结束；后续训练可略微补容量
- 已完成 < 30% 或未开始：询问换类似动作 / 直接跳过；如果是主项，默认推荐换类似动作

## 15. 失败与异常处理

Vela 不要求用户判断心理疲劳、神经疲劳或肌肉疲劳。用户只描述训练现场发生了什么。

失败类型四类：

1. 推不上去 / 没完成次数
2. 被压 / 需要保护
3. 疼痛 / 不适
4. 动作变形 / 主动放弃

提供自由输入框处理特殊情况。

### 15.1 风险等级

后台风险等级：

- 推不上去 / 没完成次数：低风险
- 动作变形 / 主动放弃：中高风险
- 被压 / 需要保护：高风险
- 疼痛 / 不适：安全事件

### 15.2 推不上去 / 没完成次数

最低风险失败类型。

处理：

- 不加重
- 延长休息
- 降低下一组目标次数
- 不一定立刻降重
- 如果连续出现，再根据目标决定降重或结束动作

### 15.3 被压 / 需要保护

高风险失败。

处理：

- 明确不建议加重
- 降重 5%-10%
- 降低目标次数
- 建议找保护
- 严重时结束主项

### 15.4 疼痛 / 不适

安全事件。

处理：

- 轻微不适可降重观察
- 明确疼痛、刺痛、关节痛、放射痛、麻、动作中疼痛加重，应停止当前动作或相关部位训练
- 整体偏保守

### 15.5 动作变形 / 主动放弃

中高风险事件。

处理：

- 轻微变形：降重、降 reps、延长休息，要求下一组动作质量优先
- 连续变形：结束当前动作，切换低风险辅助动作
- 主动放弃不羞辱用户，视为安全感不足或当前负荷不可控信号

## 16. 组间调参规则

### 16.1 连续两组轻松

不机械加重。

根据动作类型和训练目标决定：

- 主项力量动作：可小幅加重量
- 力形兼备 / 增肌目标：优先在次数区间内加 reps
- 辅项 / 孤立动作：优先增加次数、控制离心或提高动作质量
- 达到目标区间上限且动作质量良好后，再建议加重量

### 16.2 连续两组粘滞

不继续按原计划硬推。

主项力量训练：

- 优先保留重量
- 降低目标次数
- 延长休息

增肌 / 塑形 / 辅项训练：

- 优先保证动作质量和有效容量
- 必要时小幅降重

如果第三组仍粘滞或出现失败，则进入降重或结束动作逻辑。

### 16.3 加重幅度

不使用固定粗暴增量。

根据：

- 动作类型
- 器械最小单位
- 用户历史表现
- 当天状态
- 训练目标

典型策略：

- 卧推 / 上肢主项：+1.25kg 到 +2.5kg
- 深蹲：+2.5kg 到 +5kg
- 硬拉：+2.5kg 到 +5kg
- 哑铃动作：按哑铃档位
- 器械动作：按插片单位
- 孤立动作：优先加 reps / 控制，不急着加重量

如果没有合适 micro-loading 条件，不强行加重，改用增加目标次数、增加高质量组数、放慢离心、增加动作幅度、缩短休息、提高稳定性等方式进阶。

## 17. 休息计时

休息计时从用户提交本组记录后开始。

Vela 不假装自动知道用户什么时候结束一组。

休息提醒根据训练目标和状态动态调整：

- 增力 / 主项 / 冲重量：提醒柔和，允许延长
- 增肌 / 塑形 / 辅项：更强调节奏
- 上组粘滞：允许甚至建议多休息
- 上组轻松：鼓励保持节奏

计时结束时只做轻提醒：

> 可以开始下一组。

正常情况下前端不展示长文字解释。

用户可以：

- 提前结束休息
- 延长 30 秒
- 延长 60 秒

休息时间不是闹钟，而是训练变量。

## 18. 训练结束

训练不会因为完成计划自动结束。

用户需要主动点击：

> 结束训练

原则：

- 计划是可编辑的训练草图，不是流程锁
- 用户可以在训练中通过查看计划调整今日计划
- 点击结束训练后，训练立即结束，并进入训练复盘
- 不设置“完成计划后加练”的特殊流程
- 加动作、删动作、换动作、调顺序都属于今日计划动态调整

如果用户未完成计划就结束训练，Vela 不责备，只如实复盘。

## 19. 训练复盘页

训练复盘页采用：

> **结论先行，数据下钻。**

第一屏展示：

1. 一句话结论
2. 完成情况
3. 关键判断
4. 下次建议

下滑后展示：

- 详细数据
- 每个动作汇总
- 每组记录
- 训练质量
- 轻量营养建议
- 睡眠 / 恢复建议

### 19.1 信息层级

三层结构：

1. 当天最简训练总结
2. 每个动作的汇总卡
3. 点击动作卡后展开每一组的详细记录

示例：

第一层：

```text
今天完成了 Push Day 的主要训练。卧推主项完成 4 组，前两组稳定，后两组出现粘滞，因此没有继续加重。推肩提前结束，肩部容量略少。
```

第二层：

```text
卧推｜主项
4 / 4 组完成
容量：1,780 kg
体感：正常 ×2，粘滞 ×2
结论：完成稳定，但后半段接近上限
```

第三层：

```text
第 1 组
60kg × 8
体感：正常
休息：150 秒
```

### 19.2 下次建议自动写入下一次计划

训练复盘后的“下次建议”会自动更新下一次同类训练的默认计划草稿。

但它不是不可修改的命令。

用户下一次开始训练时，会看到已经被 Vela 调整过的新计划基础；随后仍需经过训练前协商，由 Vela 根据当天状态、可训练时间和不适情况再次校准。

### 19.3 不展示 e1RM 变化

训练复盘页不展示本次训练对 e1RM / 预测 1RM 的影响。

e1RM 在首页和个人数据页中动态更新。

训练复盘只关注：

- 本次训练完成情况
- 关键判断
- 下次建议
- 必要营养 / 恢复提醒
- 详细训练数据

### 19.4 轻量营养 / 睡眠 / 恢复建议

训练复盘页放轻量营养、睡眠和恢复建议，但不放在首屏。

原则：

- 轻
- 短
- 相关
- 不管理用户生活
- 不做全天饮食记录
- 不做卡路里日记
- 不做恢复 checklist

## 21. 训练模板：MVP 三分化

MVP 第一批模板采用：

> Push / Pull / Legs

原因：

- 经典
- 中阶用户熟悉
- 可以一周三练，也可以六练
- 结构清楚
- 适合主项 / 辅项预设
- 适合组间调参

### 21.1 Push Day

默认主项：

- 卧推

可替换为：

- 哑铃卧推
- 器械推胸
- 史密斯卧推
- 上斜哑铃卧推

触发替换：

- 肩部不适
- 新手
- 器械条件限制
- 缺少保护
- 用户偏好

### 21.2 Pull Day

MVP 默认不以硬拉作为 Pull Day 主项。

默认主项：

- 引体向上
- 高位下拉
- 划船类动作

硬拉作为可选主项、后链专项动作或周期性插入动作。

硬拉放 Pull Day 还是 Leg Day，本质是疲劳管理问题。

### 21.3 Leg Day

默认主项：

- 深蹲

可替换为：

- 腿举
- 哈克深蹲
- 史密斯深蹲
- 保加利亚分腿蹲

触发替换：

- 膝 / 髋 / 腰不适
- 动作基础不足
- 灵活性限制
- 器械条件限制
- 当天恢复不佳

### 21.4 动作数量

根据训练时间动态调整。

30 分钟：3-4 个动作  
45 分钟：4-5 个动作  
60 分钟：5-6 个动作

时间不够时，先砍低优先级动作，不砍热身和主项安全。

### 21.5 组数与次数

采用：

> 固定区间模板 + Vela 组间动态调整

| 动作类型 | 默认组数 | 次数区间 | 休息区间 |
|---|---:|---:|---:|
| 主项力量 | 3-5 组 | 3-8 次 | 2-4 分钟 |
| 主项增肌/力形兼备 | 3-4 组 | 6-10 次 | 90-180 秒 |
| 主要辅项 | 3-4 组 | 8-12 次 | 60-120 秒 |
| 孤立动作 | 2-4 组 | 10-20 次 | 45-90 秒 |
| 恢复/技术动作 | 2-3 组 | 8-15 次 | 60-120 秒 |

计划不是命令，计划是起点。训练中由组间反馈不断修正。

## 22. 动作替代池

MVP 内置基础动作替代池。

每个主项和常见辅项配置 3-5 个替代动作，按以下维度排序：

- 刺激相似度
- 风险水平
- 器械可得性
- 疲劳干扰

LLM 不直接自由生成替代动作，而是在结构化替代池内选择并解释。

## 23. Agent 架构原则

核心架构：

> **规则系统做底座，LLM 做交互层。**

规则 / 评分系统负责：

- 重量
- 次数
- 间歇
- 组数
- 是否结束动作
- 是否触发风险
- 是否修改计划

LLM 负责：

- 解释原因
- 用户追问
- 用户反驳
- 语言表达
- 协商改写
- 温柔但专业地说“不建议”

训练安全和关键调参不能交给 LLM 自由发挥。

## 24. Memory

Memory 是 Vela 区别于普通训练 App 的核心。

它不是聊天记忆，而是训练决策记忆。

### 24.1 ShortTermMemory

服务当前训练。

记录：

- 当前动作
- 当前组数
- 上一组体感
- 是否连续两组轻松
- 是否连续两组粘滞
- 当前动作是否出现失败
- 当前动作是否已被替换
- 今日策略：稳 / 冲 / 保
- 今日状态、时间、不适
- 当前休息是否被延长

短期记忆让 Vela 在一节训练里不失忆。

### 24.2 LongTermMemory

服务跨训练。

记录：

- 用户稳定工作组
- 用户各动作常见表现
- 用户容易粘滞的动作
- 用户反复不适的部位
- 用户被压、疼痛、动作变形等风险事件
- 用户体感刻度校准
- 用户训练偏好
- 上次复盘对下一次训练计划的更新

长期记忆让 Vela 变得越来越懂用户。

---

## 25. 核心数据记录需求

Vela Web App MVP 不是普通训练记录工具，而是训练决策 Agent。因此，系统记录数据的目的不是“留档”，而是为了支持后续的组间调参、训练复盘、下一次计划更新和长期 Memory 演化。

### 25.1 数据记录原则

第一，记录必须服务训练决策。无关训练判断的信息不进入 MVP 主链路。

第二，用户不需要填写复杂表单。系统默认预填计划值，用户只在实际情况不同、体感异常或主动调整时修改。

第三，数据要能区分：

- Vela 原计划
- 用户实际完成
- 用户主动修改
- Vela 自动调整

这四类信息不能混在一起，否则后续复盘和 Memory 会失真。

### 25.2 需要记录的信息

#### 用户画像

- 目标
- 身高体重
- 训练年限
- 伤病史
- 每周预期训练频率
- 训练偏好

冷启动阶段只强制记录：

- 目标
- 身高体重
- 训练年限
- 伤病史

#### 身体数据

- 体重
- 体脂
- 肌肉量
- 日期
- 数据来源

#### 动作库

- 动作名称
- 动作类别
- 主要训练部位
- 动作角色
- 默认组数区间
- 默认次数区间
- 默认休息区间
- 可替代动作
- 风险提示
- 器械条件

#### 训练计划

Vela 的训练计划不是静态模板，而是动态训练状态。

需要记录：

- 当前训练主题
- 今日策略
- 计划生成依据
- 今日动作列表
- 每个动作角色、组数、次数区间、休息区间、今日重点
- 训练中的动态变化
- 用户主动修改记录
- Vela 自动调整记录
- 下一次训练计划更新

计划结构应区分：

- **PlanStructure**：PPL 骨架
- **ActivePlan**：当前正在执行的动态计划
- **PlanMemory**：跨训练演化记忆

#### 单次训练记录

- 训练日期
- 训练类型
- 开始时间和结束时间
- 总时长
- 训练容量
- 完成动作数量 / 计划动作数量
- 实际完成动作列表
- 新增、删除、替换的动作
- 一句话结论
- 关键判断
- 下次建议
- 是否自动更新下一次训练计划

#### 每组训练记录

每组至少记录：

- Vela 原计划重量
- Vela 原计划次数
- Vela 原计划休息时间
- 用户实际完成重量
- 用户实际完成次数
- 用户体感状态
- 如果失败，记录失败类型
- 是否修改过重量或次数
- 是否接受 Vela 调整
- Vela 对下一组是否做了调整
- 调整内容
- 调整原因

#### 用户主动修改 vs Agent 自动调整

必须区分：

**用户主动修改**：

- 加动作
- 删动作
- 换动作
- 调整顺序
- 改重量
- 改次数
- 提前结束动作
- 结束训练
- 拒绝 Vela 建议

**Agent 自动调整**：

- 因过轻提高刺激
- 因连续轻松建议进阶
- 因连续粘滞保守调整
- 因失败降 reps / 延长休息
- 因被压降重
- 因疼痛建议停止动作
- 因器械被占推荐替代
- 因复盘更新下次训练计划

### 25.3 数据记录边界

MVP 不做：

- 全天饮食记录
- 卡路里流水账
- 复杂睡眠分期
- 医疗诊断数据
- 情绪日记
- 社交排行榜数据
- 大量动作百科学习记录

Vela 只记录能帮助训练决策的数据。

## 27. Demo 建议剧本

建议 Demo 固定为：

> **Push Day / 卧推主项训练闭环。**

展示片段：

1. 用户打开首页
2. 查看今日计划与三大项 e1RM
3. 点击开始训练
4. 分步协商状态、时间、不适
5. Vela 预判今日策略为“稳”
6. 生成 Push Day 计划
7. 用户确认计划
8. 进入卧推动作页
9. 第 1 组正常完成，不打扰，进入下一组
10. 第 2 组轻松，先记录，不大改
11. 第 3 组粘滞，记录
12. 第 4 组再次粘滞，Vela 降 reps / 延长休息
13. 用户主动结束动作或进入下一动作
14. 用户点击结束训练
15. Vela 生成复盘
16. Vela 自动更新下一次 Push Day 计划草稿

可选支线：

- 器械被占 → 替换动作
- 失败 → 失败类型确认
- 被压 → 高风险兜底
- 疼痛 → 停止当前动作
- 用户主动调整今日计划

## 28. 阶段性总结

Vela V0.3 的核心已经清晰：

> **Vela 是一个训练过程中的智能副驾。它不替用户训练，不接管用户决策，而是在训练前组织计划，在训练中观察反馈并动态调参，在异常时提供托底，在训练后复盘并更新下一次训练计划。**

Web App MVP 的目标不是做一个完整健身平台，而是证明：

> **AI 健身 Agent 可以通过轻量组后反馈，完成比传统训练记录 App 更主动、更专业、更贴近私教的训练决策闭环。**

## 1. Vela 总体产品背景

Vela 的第一性本体已经确定：

**Vela 是一个主动训练决策系统。**

它不是一个普通手环、智能表、训练记录 App、动作百科 App 或健康数据面板。它要回答的是：

1. 今天该怎么练？
2. 这一组练得怎么样？
3. 下一组该怎么调？

Vela 的核心价值是：

**稳定进步 + 少受伤。**

目标用户是中阶健身房用户，典型用户训练 6-24 个月，懂基本动作，不想长期买私教，但也不想靠网上视频和碎片攻略瞎练。他们真正缺的不是更多训练视频，也不是更多健康图表，而是一个能结合当天状态、历史训练和当下表现，告诉他“今天该怎么练、下一组怎么调”的训练 Agent。

## 2. Vela App 主线已经确定

当前主线是 Web App MVP。

App MVP 的定位：

**先用 Web App 跑通 AI 健身 Agent 的训练日闭环。**

核心流程是：

首页 → 开始训练 → 训练前协商 → 今日计划 → 动作训练页 → 组后记录 → Agent 组间调参 → 用户主动结束训练 → 训练复盘 → 自动更新下一次训练计划草稿

App MVP 的重点不是完整健身平台，而是证明：

**AI 健身 Agent 可以通过轻量组后反馈，完成比传统训练记录 App 更主动、更专业、更贴近私教的训练决策闭环。**

App 里的关键训练交互原则：

- 用户拥有训练主导权，Vela 拥有专业建议权。
- Vela 是智能副驾，不是训练控制器。
- 正常时隐身，异常时介入。
- 正常完成不弹建议卡、不解释，直接按计划进入下一组。
- 体感前台词为：过轻 / 轻松 / 正常 / 粘滞 / 失败。
- 失败不要求用户判断心理、神经、肌肉疲劳，只问现场事件：推不上去/没完成次数，被压/需要保护，疼痛/不适，动作变形/主动放弃。
- 休息计时从用户提交组后记录开始。
- 训练不自动结束，用户主动点击结束训练。
- 复盘页结论先行，数据下钻。
- 复盘后的下次建议会自动更新下一次同类训练计划草稿，但下次训练当天仍会再协商状态、时间和不适。

App 的存在对硬件讨论很重要：

**硬件不是产品本体，而是给 Vela Agent 提供更高质量现场数据的数据入口。**

## 3. 当前硬件三叉路口

现在进入一个新的硬件产品定义阶段。

已讨论过的硬件可能性包括：

1. 腕带 / 手环 / 手镯  
   用于全天候身体状态、心率、HRV、睡眠、提醒、小屏确认等。

2. 手套 + 多点压力传感器  
   对上肢力量动作可能很强，可以测握持压力、推拉方向、左右差异，但有清洗、佩戴、睡眠/久坐无法兼顾、下肢动作兼容性差等问题。

3. 手机摄像头 / 视觉识别  
   适合动作姿态辅助判断，但需要用户架手机，可能拍到别人，隐私和摩擦较高。

4. 当前重点探索方向：Vela Wrap / Sensor Wrap  
   一种类似宽版卷尺、柔性护腕、可卷绕绑带的形态，可以包裹在杠铃、哑铃、器械把手、高位下拉杆、坐姿划船把手等器械接触点上。

当前新对话应主要讨论第 4 个方向：**Vela Wrap / Sensor Wrap**。

## 4. Vela Wrap 当前概念

暂定命名：

- Vela Wrap
- Vela Sensor Wrap
- Vela Training Wrap
- Vela Force Wrap
- 中文可暂称：训练感应带 / 力感带 / 卷绕式训练传感器 / 器械传感带

当前概念定义：

**Vela Wrap 是一条可卷绕训练器械传感带，包裹在杠铃、哑铃或器械把手上，通过压力阵列和 IMU 捕捉训练中的接触压力、左右压力差、动作节奏和速度趋势，为 Vela Agent 的组间调参提供更高质量的现场数据。**

它的产品定位不应是：

- 精准测重量的电子秤
- 精准测绝对功率的实验室仪器
- 完整动作纠错系统
- 所有器械通用传感器

更稳的定位是：

**让普通器械拥有“接触感知 + 运动感知”的临时传感皮肤。**

或者：

**训练接触质量 + 运动节奏传感器。**

## 5. 当前技术可行性判断

### 5.1 压力传感器能做什么

压力传感器 / 柔性压力阵列 / FSR 更适合测：

- 接触压力
- 握持压力
- 左右压力差
- 压力变化趋势
- 压力峰值变化
- 接触稳定性
- 是否出现明显偏侧
- 推 / 拉过程中压力是否波动变大
- 疲劳后左右压力差是否扩大

注意：它测的是 **接触压力差**，不等于严格的左右肌力差。

前台表达要谨慎：

可以说：

- 左右接触压力是否均衡
- 左右发力趋势是否同步
- 推/拉过程中是否出现明显偏侧
- 接触稳定性是否下降

不建议说：

- 精准测左右肌力
- 精准判断哪边肌肉发力不足

### 5.2 压力不能可靠推断加速度

不要走这条链路：

**压力差 → 加速度 → 功率**

这条链路不稳，误差会叠加。

加速度应该由 IMU 直接测。

### 5.3 IMU 应该是必要传感器

建议 Vela Wrap 至少包含：

- 加速度计
- 陀螺仪
- 可能的磁力计可选

IMU 用于：

- 重复次数
- 节奏
- 离心 / 向心阶段
- 速度趋势
- 速度衰减
- 粘滞点附近速度下降
- 相对功率估算的运动数据部分

### 5.4 功率估算的正确路径

功率大致需要：

**力 × 速度**

Vela Wrap 如果要做功率，较合理路径是：

1. 用户或 App 已知本组重量，例如卧推 60kg。
2. Wrap 的 IMU 测出每次动作的速度/加速度趋势。
3. 系统估算每次 rep 的速度、速度衰减、相对功率趋势。
4. 压力传感器辅助判断左右稳定性和接触质量。

即：

**功率趋势估算 = 已知负重 + IMU 运动数据。**  
**压力数据 = 辅助解释稳定性和左右差。**

不要说：

**压力传感器直接测重量 / 直接测功率。**

### 5.5 能实现什么

比较可行：

- 左右接触压力差
- 握持稳定性
- 压力峰值变化
- 推 / 拉过程压力波动
- 重复次数
- 动作节奏
- 离心 / 向心阶段粗略识别
- 速度趋势，前提是加 IMU
- 疲劳导致的速度下降趋势，前提是 IMU 稳
- “这一组比上一组更不稳”这种相对判断

需要谨慎：

- 左右发力均衡
- 功率趋势
- 深蹲上背杠位偏移
- 高位下拉 / 划船左右同步性

不建议第一版承诺：

- 精准测真实重量
- 精准测绝对功率
- 完整动作纠错
- 判断所有器械动作质量
- 替代摄像头判断姿态
- 替代测力台 / 专业 VBT 设备

## 6. 推荐的首版传感器组合

### 6.1 柔性压力传感阵列 / 多点 FSR

用途：

- 左右压力差
- 握持/接触压力分布
- 推拉过程压力变化
- 压力稳定性
- 是否出现明显偏侧

局限：

- FSR 便宜、薄、容易集成
- 但精度、线性、漂移、重复性都有限
- 更适合趋势和分布，不适合高精度力学测量

### 6.2 IMU

用途：

- 重复次数
- 节奏
- 离心 / 向心阶段
- 速度趋势
- 速度衰减
- 相对功率估算

这个是必须的。没有 IMU，Wrap 很难把加速度、速度、功率讲圆。

### 6.3 蓝牙、电池与固定结构

必须考虑：

- 蓝牙通信
- 小型电池
- 充电方式
- 防汗
- 耐磨
- 清洁
- 柔性封装
- 魔术贴 / 磁吸 / 卡扣 / 卷尺式回收等固定方式

真正难点不只是传感器，而是：

- 绑得稳不稳
- 不同杆径能否适配
- 压力传感器有没有滑移
- 汗液和摩擦会不会影响读数
- 用户每次绑的位置是否一致
- 杠铃滚花会不会磨损表面
- 器械把手形状差异太大
- 深蹲时上背接触区域是否能稳定固定

## 7. 建议优先验证场景

不要从深蹲开始。深蹲上背那个想法有趣，但变量太多。

推荐顺序：

### 场景 1：高位下拉 / 坐姿划船

原因：

- 左右手握杆或把手
- 拉力趋势明显
- 器械轨迹相对稳定
- 安全风险低
- 压力差更容易解释

验证目标：

- 左右压力差能不能稳定反映左右拉动趋势
- 疲劳时压力波动是否变大
- 能不能识别 reps 和节奏

### 场景 2：卧推

原因：

- 和 Vela App 主 Demo 强相关
- 适合展示 Agent 调参
- 但杠铃会旋转，手腕角度会变化，握持压力不等于推力

验证目标：

- 左右握持压力是否稳定
- 最后几次是否压力波动增大
- IMU 能不能测出推起速度下降
- Vela 能不能据此判断“下一组不建议加重”

### 场景 3：哑铃动作

如果每只哑铃一个 Wrap，左右对比更直观。  
如果只有一个 Wrap，就不适合做左右均衡。

### 场景 4：深蹲上背接触

作为概念探索，不作为第一轮核心验证。

可以验证：

- 杠位是否偏移
- 左右接触压力是否明显不均

不要承诺：

- 判断完整深蹲动作质量

## 8. Wrap 给 App 的数据接口假设

Wrap 不应该单独卖“数据”，它的数据要服务 Vela Agent 的组间调参。

可能输出字段：

- left_pressure_avg
- right_pressure_avg
- pressure_asymmetry
- pressure_stability
- peak_pressure_delta
- rep_count
- rep_tempo
- eccentric_tempo
- concentric_tempo
- concentric_velocity_trend
- velocity_loss_percent
- contact_shift
- confidence_score

App 不应该直接说：

“你右胸发力弱。”

而应该说：

“这组右侧压力波动更大，最后两次推起速度下降明显。下一组不建议加重，先保持重量，把左右稳定性做稳。”

---

## 9. Wrap 与 App 的关系

App 是主线，Wrap 是增强数据入口。

App 负责：

- 训练计划
- 训练前协商
- 组后记录
- 组间调参
- 训练复盘
- Memory
- 下一次计划更新

Wrap 负责：

- 提供更高质量现场数据
- 减少用户手动反馈负担
- 识别接触压力异常
- 识别速度/节奏趋势
- 辅助判断左右不均、疲劳、粘滞和不稳定

Wrap 不能变成另一个孤立硬件。它必须服务：

**Vela Agent 下一组怎么调。**

## 11. 下一轮硬件产品定义要解决的问题

请在新对话中继续使用共创方式，一次问 1-3 个问题，不要一次性把全部问题都问完。

建议从以下问题开始：

### 第一阶段：产品定位与边界

1. Vela Wrap 的一句话定义是什么？
2. 它是训练辅助设备、传感器配件，还是 Vela Pro 系列硬件？
3. 它第一版到底解决什么问题？
4. 它不解决什么问题？
5. 它和腕带 / 手套 / 摄像头相比，核心差异是什么？

### 第二阶段：形态定义

1. 它像宽版卷尺、护腕、绑带，还是柔性贴片？
2. 它如何固定在杠铃 / 哑铃 / 器械把手上？
3. 它是否需要自回卷结构？
4. 它是否用魔术贴、磁吸、卡扣、硅胶摩擦还是弹性卷绕？
5. 它的尺寸、宽度、厚度大概是什么？

### 第三阶段：传感器定义

1. 必须有哪些传感器？
2. 压力阵列怎么布局？
3. IMU 放在哪里？
4. 是否需要双侧传感区？
5. 是否需要校准？
6. 是否需要测杆径 / 包裹紧度？

### 第四阶段：首批训练场景

1. 第一版先验证高位下拉、坐姿划船、卧推，还是哑铃？
2. 哪些动作不能第一版承诺？
3. 深蹲上背场景放在哪个阶段？
4. 每个动作具体测什么？

### 第五阶段：数据解释

1. 压力差如何转成用户能理解的结论？
2. IMU 速度趋势如何辅助调参？
3. 哪些指标给用户看，哪些只给 Agent 后台用？
4. 如何避免伪精确？

### 第六阶段：App 联动

1. Wrap 在 App 中如何显示连接状态？
2. Wrap 数据如何出现在动作训练页？
3. 正常时是否隐藏数据？
4. 异常时如何提示？
5. 训练复盘中如何体现 Wrap 贡献？

### 第七阶段：MVP 验证

1. 一个月内能不能做物理原型？
2. 如果不能做真硬件，能不能做概念渲染 + 模拟数据接入？
3. 最小实验怎么做？
4. 需要哪些材料和传感器模块？
5. 验证成功标准是什么？

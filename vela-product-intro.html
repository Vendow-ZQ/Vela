Vela：从主动决策系统到智能硬件。

Vela 的产品定义是一个 AI 健身 Agent：以训练决策为本体，先通过 Web App 与 Agent 本体跑通训练日闭环，再通过 Sensor Wrap、Band 与视觉系统逐步获得更高质量的训练现场和身体状态数据。

战略判断机会不在“更多健康数据”，而在把数据转成下一步训练动作。
阶段一判断先用 Web App + Agent 本体跑通 Push Day / 卧推组间调参闭环。
硬件判断硬件不是本体，是给 Agent 提供更高质量现场数据的入口。
边界判断不做伪精确，不承诺精准测重量、绝对功率或完整动作纠错。
00

执行结论：先给产品判断。

Research 给出的市场机会是 AI健身、数字健身、可穿戴AI 的高增速；Product Definition 给出的产品答案是“主动训练决策系统”；Hardware Wrap Context 给出的硬件原则是“服务组间调参，不单独卖数据”。

一句话产品定义

Vela 是一个 AI 健身 Agent：基于用户的生理数据、训练数据和实时训练质量，主动判断今天怎么练、这一组练得怎么样、下一组怎么调。

一级产品本体

不是腕带、手表、耳机、摄像头、体脂秤或训练记录 App，而是主动训练决策系统。硬件只是数据入口或体验增强器。

首个验证闭环

Web App MVP 聚焦中阶健身房用户，以 Push Day / 卧推为 Demo 主线，验证训练前协商、组后反馈、组间调参、训练复盘和下次计划更新。

核心产品判断：Vela 不以“健康穿戴”作为第一性定义，而以“训练决策”作为第一性定义。产品价值不是展示更多数据，而是把数据转化为今天怎么练、这一组练得怎么样、下一组怎么调。

02

用户画像与痛点：他们不是缺数据，是缺行动。

Research 反复指向同一个问题：可穿戴和健身 App 已经能记录很多数据，但用户真正愿意付费的是“今天怎么练、下一组怎么调、怎么避免受伤”。

A. 数据驱动运动增强

25-45 岁，男性偏多，中高收入，3年以上运动习惯。需要高精度数据、训练负荷分析和恢复建议，适合成为 Vela Pro / Wrap 的早期种子用户。

B. 康复 / 疾病预防

对健康数据敏感，但功能复杂时高流失。不是 MVP 主战场，可作为未来 Band 与健康托底线延展。

C. 体态 / 体型塑造

18-35 岁，新手更多，目标是减脂、增肌、体态改善。需要低门槛计划和持续反馈，但训练决策深度较低。

弃用原因典型反馈本质问题Vela 应对
“So what?”问题Whoop 告诉我恢复率 34%，那我今天要不要去健身房？数据没有转成训练动作。输出下一组 / 今日策略，而不是单独展示分数。
订阅价值不匹配第一年后 $30/月，Apple Watch 免费能做 80%。续费节点价值不可感知。6个月内让用户看到训练表现或伤病风险管理收益。
佩戴摩擦$500 买的 Athos，洗4次后就不穿了。硬件日常维护成本过高。App 先行；硬件必须低摩擦，Wrap 只在训练时使用。
数据信任崩塌它说我睡了8小时，我其实2点才睡。一次错误会毁掉长期信任。前台表达保守，避免伪精确；保留用户修正权。
健康焦虑为了睡眠分数反而睡不好。指标设计会反噬用户。不做焦虑化健康报告，包装为训练质量和下一步建议。
03

竞品格局：每类产品都缺一块。

Whoop / Oura 有传感器和订阅，但缺主动训练指导；Garmin / Apple 有生态，但健身深度或 AI 深度不足；Future 解决了行动建议但不能规模化；Vi 证明了“耳机+AI健身”不能只靠语音聊天。

04

失败案例：Vela 要主动避开的坑。

失败不是“市场不存在”，而是产品把精度、硬件、内容、AI 或语音能力放错了位置。Vela 的风险控制必须从定义阶段就写进边界。

Athos 智能运动服

  • EMG 传感器编织进压缩衣裤，套装约 $500。
  • 失败原因：清洗/充电/穿戴摩擦、数据解读门槛、硬件成本。
  • 教训：精度不等于用户价值；硬件摩擦会杀死训练习惯。

Mirror / Lululemon Studio

  • 2020 年 $5 亿收购，2023 年官方停产。
  • 失败原因：COVID 红利消退、高价硬件、内容护城河不足。
  • 教训：不要依赖居家健身叙事，聚焦训练表现提升。

Halo Sport 神经刺激耳机

  • tDCS 经颅电刺激耳机，消费市场教育成本极高。
  • 失败原因：科学证据争议、监管模糊、效果不可感知。
  • 教训：消费级生物技术必须可解释、可感知、可信。

Vi AI健身耳机

  • 与 Shokz 形态最接近的失败先例。
  • 失败原因：语音 AI 体验不足，运动中实时需求变化太复杂。
  • 教训：不能只做“加了AI的耳机”，必须有窄场景闭环。
05

产品哲学：主动指导，而不是被动监测。

Vela 的长期价值不是让用户每次训练都“爽”，而是稳定进步、少受伤、少瞎练。产品必须克制：正常时不打扰，异常时才介入。

主动指导是核心

睡眠、HRV、体脂、体重、恢复状态，不是为了健康大屏，而是为了训练决策。

稳定进步 + 少受伤

从增肌、塑形、减脂、增力诉求进入，长期留存靠更科学训练。

正常时隐身

按计划完成时不弹建议卡、不解释、不打扰，直接进入下一组。

异常时介入

连续粘滞、失败、疼痛、被压、动作变形时，明确调整或反对高风险行为。

06

产品定义：Vela 的本体不是硬件。

硬件、传感器、第三方健康数据、体测报告、摄像头、腕带、Wrap / Sensor，都是主动训练决策系统的数据入口或体验增强器。

Vela 不是什么

  • 不是健康数据 App。
  • 不是训练记录 App。
  • 不是动作百科 App。
  • 不是先从腕带、手表、手套或摄像头定义产品。
  • 不是全能健康管理、医疗诊断或营养记录平台。

Vela 是什么

  • 一个主动训练决策系统。
  • 一个能回答“今天怎么练、这一组怎样、下一组怎么调”的 AI 健身 Agent。
  • 一个以 Web App 先跑通训练闭环，再接入硬件增强的产品路线。
  • 一个规则系统做底座、LLM 做交互层的智能副驾。
07

Agent 边界：用户主导，Vela 建议。

Vela 是辅助型智能体，不是训练控制器。安全和关键调参不能交给 LLM 自由发挥，必须由规则系统兜底。

用户决定Vela 负责LLM 负责规则系统负责
是否开始训练、执行计划、修改动作、接受建议、结束动作、结束训练。 生成默认计划、组间调参建议、识别风险、提供替代方案、记录训练变化。 解释原因、协商改写、回答追问、温柔但专业地说“不建议”。 重量、次数、间歇、组数、风险触发、是否结束动作、是否修改计划。

可建议

根据历史工作组、当天状态、组后反馈，建议保持、加 reps、小幅加重、延长休息、替换动作。

可反对

疼痛、被压、明显动作变形或高风险坚持冲重量时，Vela 应明确反对当前动作策略。

不可越界

不做医疗诊断、不做激素/药物建议、不伪装绝对测力设备、不用 LLM 自由决定安全动作。

08

运行逻辑：训练日闭环，而不是聊天机器人。

MVP 的核心 Demo 固定为 Push Day / 卧推主项。目标不是做完整平台，而是证明轻量反馈可以驱动可信调参。

01首页：今日建议与计划入口
02训练前协商：状态 / 时间 / 不适
03今日计划：稳 / 冲 / 保策略
04动作执行：当前组聚焦
05组后反馈：过轻 / 轻松 / 正常 / 粘滞 / 失败
06复盘：结论先行，更新下次草稿
体感状态前台含义调参策略是否打扰
过轻计划明显低估强度立即提高训练刺激，可加 reps 或小幅加重立即介入
轻松有余力，但不确定是否稳定先观察，连续两组轻松再调整观察
正常计划匹配继续原计划不打扰
粘滞完成了，但最后几次明显卡住先观察，连续两组后保守调整观察
失败未完成或出现安全事件确认失败类型，降 reps / 降重量 / 停止动作 / 替代动作立即介入
09

硬件形态：App 是主线，Wrap 是增强入口。

当前硬件探索重点是 Vela Wrap / Sensor Wrap：可卷绕训练器械传感带，把普通杠铃、哑铃和器械把手临时变成可感知的训练接触界面。

一句话硬件定义

Vela Wrap 是一条可卷绕训练器械传感带，包裹在杠铃、哑铃或器械把手上，通过压力阵列和 IMU 捕捉接触压力、左右压力差、动作节奏和速度趋势，为 Vela Agent 的组间调参提供更高质量现场数据。

不应承诺

  • 精准测真实重量。
  • 精准测绝对功率。
  • 完整动作纠错。
  • 判断所有器械动作质量。
  • 替代摄像头、测力台或专业 VBT 设备。
硬件方向优势问题当前判断
腕带 / 手环全天候心率、HRV、睡眠、提醒、小屏确认。容易落入 Whoop / Apple / Garmin 竞争。V2 身体状态入口，不是本体。
手套 + 压力传感上肢动作压力数据强。清洗、佩戴、下肢兼容性差。风险高，暂不主推。
手机摄像头 / 视觉适合动作姿态和辅助判断。架手机、隐私、拍到他人、健身房摩擦。未来增强，不做 MVP 主链路。
Sensor Wrap训练时使用，包裹器械，场景聚焦。固定、滑移、汗液、杆径适配、读数一致性。当前重点探索方向。
10

技术与材料选型：先追求可靠趋势，不追求实验室精度。

Wrap 的技术路线应围绕“可解释的相对变化”设计：压力阵列看接触质量和左右趋势，IMU 看 reps、节奏和速度趋势，App / Agent 把这些转成调参建议。

传感器组合

  • 柔性压力传感阵列 / 多点 FSR。
  • IMU：加速度计 + 陀螺仪，磁力计可选。
  • 蓝牙通信，小型电池，训练中低延迟连接。

材料方向

  • 外层：耐磨防汗 TPU / 硅胶包覆。
  • 接触层：高摩擦防滑纹理，适配滚花和器械把手。
  • 结构：魔术贴 / 磁吸 / 卡扣 / 弹性卷绕待验证。

工程难点

  • 不同杆径适配。
  • 压力传感器滑移。
  • 汗液、摩擦、滚花磨损。
  • 每次绑的位置一致性。
模块可做什么不能说什么前台表达
FSR 压力阵列接触压力、左右压力差、压力稳定性、压力峰值变化。精准测左右肌力、精准判断哪边肌肉发力不足。“左右接触压力是否均衡”“接触稳定性是否下降”。
IMUreps、节奏、离心/向心阶段、速度趋势、速度衰减。没有已知负重时直接测功率。“最后几次推起速度下降明显”。
App 已知负重结合 IMU 做相对功率趋势估算。绝对功率精准测量。“这组相对输出趋势下降，不建议加重”。
Confidence Score判断本组数据是否可用于调参。隐藏不确定性。低置信度时退回用户体感反馈。
11

算法边界：安全、调参和伪精确是三条红线。

算法系统必须把“可测、可推断、不可承诺”分开。Agent 对用户说的是训练建议，不是传感器炫技，也不是医学结论。

可进入 Agent 后台的字段

  • left_pressure_avg / right_pressure_avg
  • pressure_asymmetry / pressure_stability
  • peak_pressure_delta / contact_shift
  • rep_count / rep_tempo
  • eccentric_tempo / concentric_tempo
  • concentric_velocity_trend / velocity_loss_percent
  • confidence_score

前台应该如何翻译

不要说“你右胸发力弱”。应该说：“这组右侧压力波动更大，最后两次推起速度下降明显。下一组不建议加重，先保持重量，把左右稳定性做稳。”

算法层职责边界失败兜底
规则引擎决定重量、次数、休息、风险触发和动作是否继续。必须可审计，不交给 LLM 自由发挥。风险高时保守调整。
LLM 交互层解释原因、处理用户追问、协商改写计划。不能单独决定安全和关键训练变量。调用规则建议，不确定时请求确认。
ShortTermMemory记录当前训练中的连续轻松、连续粘滞、失败、替换和休息延长。仅服务当前训练。丢失时回到本组记录和用户确认。
LongTermMemory记录稳定工作组、常见短板、不适部位、训练偏好、上次复盘更新。不做医疗病史推断。用户可修改，Agent 需记录修改来源。
12

路线图与阶段定义。

资源投入顺序建议从“Agent 闭环先行，硬件验证并行”开始。先证明用户愿意让 Vela 参与下一组训练决策，再验证智能硬件是否能提升判断精度和反馈可信度。

阶段目标交付物成功标准
阶段一：Web App + Agent 本体跑通训练日闭环和主动训练决策。首页、训练前协商、今日计划、动作页、组后反馈、组间调参、复盘页、Memory、Agent 规则表、Mock 数据。用户理解 Vela 是 AI健身Agent，并愿意每组后做轻量反馈；系统能给出可信的下一组建议。
V1：Vela Sensor Wrap提供训练现场质量数据。压力阵列、IMU、BLE、固定结构、模拟数据接口、首批动作验证场景。能辅助判断不稳定、速度下降、左右压力波动，并提升组间调参可信度。
V2：Vela Band提供全天候身体状态入口。心率、HRV、睡眠、休息提醒、震动反馈、小屏确认、第三方健康数据联动。增强训练前状态判断、组间节奏提醒和日常恢复托底，但不替代 Agent 主线。

阶段决策 1

阶段一是否明确以 Web App + Agent 本体为核心，而不是以硬件单品作为第一入口？

阶段决策 2

V1 是否将 Sensor Wrap 定义为训练现场数据入口，而不是独立智能健身硬件品类？

阶段决策 3

第一版算法边界是否明确为：不做精准测重、绝对功率、完整动作纠错和医疗诊断？
